# 1 Введение

В рамках лабораторной работы мы реализовали **систему авторизации и разграничения прав доступа** для веб-приложения "Библиотека". Документ описывает процесс парного программирования, результаты работы и выводы.

**Технический стек:**
- Бэкенд: Java + Spring Boot + JPA + PostgreSQL
- Фронтенд: React + Material-UI + Axios

## 2.2 Этапы разработки

| Итерация | Направление | Действия |
|:---|:---|:---|
| **1** | Бэкенд → Фронтенд | • Бэкенд-разработчик создал эндпоинты для регистрации и логина (`/api/auth/register`, `/api/auth/login`)<br>• Бэкенд-разработчик написал тесты для проверки этих эндпоинтов<br>• Фронтенд-разработчик реализовал компоненты `LoginPage` и `RegisterPage`, подключил их к API |
| **2** | Фронтенд → Бэкенд | • Фронтенд-разработчик создал компонент `AdminRoute` для защиты админских маршрутов<br>• Бэкенд-разработчик добавил поле `role` в ответ при логине и реализовал проверку прав на сервере |
| **3** | Бэкенд → Фронтенд | • Бэкенд-разработчик добавил эндпоинты для книг и авторов с проверкой ролей (`@PreAuthorize`)<br>• Фронтенд-разработчик реализовал отображение кнопок управления только для админа с помощью `useAuth().isAdmin` |
| **4** | Фронтенд → Бэкенд | • Фронтенд-разработчик создал компонент `BookDetails` с возможностью оставлять отзывы<br>• Бэкенд-разработчик добавил эндпоинты для отзывов и проверку, что пользователь может редактировать только свои отзывы |

<a name="positive_experience"/>

## 3.2 Положительные впечатления

1. **Обмен знаниями** – бэкенд-разработчик узнал, как работает React-хуки и контекст, фронтенд-разработчик лучше разобрался в Spring Security
2. **Скорость разработки** – справились быстрее, чем если бы делали по отдельности, так как не тратили время на согласование API
3. **Качество кода** – код проходил двойную проверку сразу в процессе написания
4. **Меньше багов** – благодаря постоянному переключению ролей находили ошибки на ранних этапах
5. **Интересный опыт** – необычно и увлекательно работать в таком режиме

<a name="problems"/>

# 4 Проблемы и их решения

| Проблема | Решение |
|:---|:---|
| Сложности с настройкой окружения (разные IDE, разные версии Java/Node) | Создали общий документ с инструкцией и договорились использовать одинаковые версии |
| Ожидание партнера (один ждал, пока другой закончит свою часть) | Научились параллелить задачи – пока один ждал, писал документацию или тесты |
| Задачи разного размера (в стиле "пинг-понг" сложно работать с задачами разного объема) | Старались разбивать задачи на примерно равные по времени итерации |
| Несоответствие типов данных при интеграции | Добавили автоматическую генерацию TypeScript-типов из OpenAPI спецификации |
| "Залипание" за клавиатурой (забывали менять роли) | Поставили таймер на 30 минут для смены "ведущего" |

<a name="prospects"/>

# 5 Перспективы использования ПП

## 5.1 Типы задач для парного программирования

| Тип задачи | Обоснование |
|:---|:---|
| Сложные архитектурные решения | Когда нужно спроектировать новую фичу, лучше делать это в паре |
| Разработка критически важных компонентов | Например, платежная система, где ошибки дорого обходятся |
| Онбординг новых разработчиков | Чтобы быстрее ввести в курс дела |
| Решение сложных багов | Особенно когда один разработчик уже "замылил глаз" |
| Написание сложных SQL-запросов | Для оптимизации производительности |

## 5.2 Предпочитаемые стили для разных ситуаций

| Ситуация | Стиль |
|:---|:---|
| Сложные задачи | "На равных" или "Ведущий-ведомый" (в зависимости от опыта) |
| Разработка новой фичи | "Пинг-понг" (особенно если фича требует синхронных изменений в бэкенде и фронтенде) |
| Код-ревью | Элементы ПП, чтобы сразу обсуждать изменения, а не писать длинные комментарии в Pull Request |

<a name="conclusions"/>

# 6 Выводы

Парное программирование оказалось **эффективным инструментом**, особенно для задач, требующих тесной интеграции между бэкендом и фронтендом. Стиль **"Пинг-понг"** идеально подошел для нашей задачи, так как позволил синхронно развивать обе части приложения.

## Плюсы и минусы

| ✅ Плюсы | ❌ Минусы |
|:---|:---|
| Меньше ошибок | Требует синхронизации графиков работы |
| Быстрое решение проблем | Нужно привыкнуть к стилю общения |
| Обмен знаниями | Устаешь быстрее, чем при соло-работе |
| Высокая вовлеченность в процесс | |

В целом, опыт оказался очень полезным, и мы планируем применять парное программирование в будущих проектах!
