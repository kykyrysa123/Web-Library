# 1 Введение

<a name="appointment"/>

## 1.1 Назначение
В этом документе описаны функциональные и нефункциональные требования к Telegram-боту **GiftBot**, предназначенному для оформления заказов товаров из Китая. Документ ориентирован на команду разработки и тестирования приложения.

<a name="business_requirements"/>

## 1.2 Бизнес-требования

<a name="initial_data"/>

### 1.2.1 Исходные данные
С ростом популярности китайских маркетплейсов пользователи сталкиваются с трудностями при оформлении заказов: языковой барьер, сложности с доставкой и оплатой. Существующие сервисы часто имеют сложный интерфейс и высокие комиссии.

<a name="business_opportunities"/>

### 1.2.2 Возможности бизнеса
Приложение позволит:
- Упростить процесс заказа товаров из Китая через Telegram
- Автоматизировать сбор информации о заказах
- Предоставить удобный интерфейс для отслеживания статуса заказов
- Обеспечить администраторам полный контроль над заказами

<a name="project_boundary"/>

### 1.2.3 Границы проекта
**GiftBot** позволяет пользователям:
- Оформлять заказы с китайских маркетплейсов
- Отправлять скриншоты и ссылки на товары
- Отслеживать статус заказов

Администраторы могут:
- Управлять всеми заказами
- Одобрять или отклонять заказы
- Экспортировать данные в Excel
- Управлять списком поддерживаемых сайтов

Исключено:
- Прямая оплата через бота
- Автоматическая проверка наличия товаров
- Интеграция с системами доставки

<a name="analogues"/>

## 1.3 Аналоги
Аналогичные сервисы: посреднические услуги заказа из Китая, сервисы доставки из Китая.  
Преимущества **GiftBot**: интеграция с Telegram, простой интерфейс, автоматизация сбора данных, специализация на китайских маркетплейсах.

<a name="user_requirements"/>

# 2 Требования пользователя

<a name="software_interfaces"/>

## 2.1 Программные интерфейсы
Приложение использует:
- Telegram Bot API для взаимодействия с пользователями
- SQLite для хранения данных о заказах
- Java Spring Boot 3.1.4 для backend-логики
- Apache POI для генерации Excel-отчетов
- Spring Security для управления доступом

<a name="user_interface"/>

## 2.2 Интерфейс пользователя
Интерфейс реализован через Telegram-бота и включает:
- Интерактивные клавиатуры с кнопками
- Пошаговые формы оформления заказа
- Возможность отправки фотографий и текста
- Меню с основными командами (/start, /order, /status, /sites, /admin)

<a name="user_specifications"/>

## 2.3 Характеристики пользователей

<a name="user_classes"/>

### 2.3.1 Классы пользователей

| Класс пользователей | Описание |
|:---|:---|
| Анонимные пользователи | Могут просматривать информацию о боте, список сайтов, информацию об админе |
| Зарегистрированные пользователи | Могут оформлять заказы, отслеживать статус, управлять своими данными |
| Администраторы | Полный доступ к управлению заказами, пользователями, настройкам системы |

<a name="application_audience"/>

### 2.3.2 Аудитория приложения

<a name="target_audience"/>

#### 2.3.2.1 Целевая аудитория
Пользователи, желающие заказывать товары из китайских маркетплейсов с доставкой в Беларусь и другие страны СНГ.

<a name="collateral_audience"/>

#### 2.3.2.2 Побочная аудитория
Мелкие предприниматели, занимающиеся перепродажей товаров из Китая.

<a name="assumptions_and_dependencies"/>

## 2.4 Предположения и зависимости
1. Наличие аккаунта Telegram у пользователя
2. Стабильное интернет-соединение
3. Доступ к китайским маркетплейсам
4. Наличие администратора для обработки заказов

# 3 Системные требования

<a name="functional_requirements"/>

## 3.1 Функциональные требования

<a name="main_functions"/>

### 3.1.1 Основные функции

<a name="user_registration"/>

#### 3.1.1.1 Регистрация и начало работы
| Функция | Требования | 
|:---|:---|

<a name="order_creation"/>

#### 3.1.1.2 Оформление заказа
| Функция | Требования | 
|:---|:---|
| Ввод данных заказа | Прием ссылок, описаний товаров, скриншотов |
| Пошаговое оформление | Последовательный сбор информации о товаре, пользователе, доставке |

| Функция | Требования |

|:---|:---|

| Инструкции | Информация о процессе заказа |

<a name="admin_management"/>

#### 3.1.1.5 Административное управление
| Функция | Требования | 
|:---|:---|
| Просмотр всех заказов | Полный список заказов с фильтрацией |
| Экспорт в Excel | Генерация отчетов по заказам |
| Управление сайтами | Добавление и редактирование списка поддерживаемых площадок |

<a name="restrictions_and_exclusions"/>

### 3.1.2 Ограничения и исключения
1. Бот не обрабатывает платежи
2. Бот не проверяет доступность товаров в реальном времени
3. Бот не интегрирован с системами логистики
4. Требуется ручная обработка заказов администратором

<a name="non-functional_requirements"/>

## 3.2 Нефункциональные требования

<a name="quality_attributes"/>

### 3.2.1 Атрибуты качества

<a name="requirements_for_ease_of_use"/>

#### 3.2.1.1 Требования к удобству использования
- Интуитивно понятный интерфейс на основе кнопок
- Пошаговые инструкции для пользователей
- Минимальное количество действий для оформления заказа
- Поддержка русского языка

<a name="security_requirements"/>

#### 3.2.1.2 Требования к безопасности
- Разграничение прав доступа между пользователями и администраторами
- Валидация вводимых данных
- Защита от спама и злоупотреблений
- Безопасное хранение пользовательских данных

<a name="performance_requirements"/>

#### 3.2.1.3 Требования к производительности
- Время отклика бота не более 2 секунд
- Поддержка одновременной работы multiple пользователей
- Стабильная работа при высокой нагрузке

<a name="external_interfaces"/>

### 3.2.2 Внешние интерфейсы
- Telegram Bot API для коммуникации
- SQLite для хранения данных
- Java Spring Boot для backend-логики

<a name="restrictions"/>

### 3.2.3 Ограничения
- Зависимость от Telegram API
- Ограничения Telegram на размер сообщений и файлов
- Необходимость ручной модерации заказов
- Ограниченная функциональность мобильных устройств
