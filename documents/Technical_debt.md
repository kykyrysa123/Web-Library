### <a name="intro"/>1. Введение
В рамках лабораторной работы был проинспектирован проект **«Веб-библиотека»** — веб-приложение для управления книгами и авторами с системой авторизации и разграничением прав доступа (администратор/пользователь).

**Технический стек:**
- **Бэкенд:** Java 17, Spring Boot 3.2.4, Spring Security, JPA, PostgreSQL, Maven
- **Фронтенд:** React 18, Material-UI, React Router, Axios

**Цель работы:** выявить признаки технического долга в проекте, разработать план его устранения, оценить трудозатраты и принять решение о необходимости рефакторинга.

### <a name="inspection"/>2. Инспекция проекта на наличие технического долга
Анализ проводился на основе предоставленного кода, логов ошибок и общей архитектуры проекта.

#### <a name="unreadable_code"/>2.1 Непонятный / нечитабельный код
| Компонент | Проблема | Пример |
| :--- | :--- | :--- |
| `BookDetails.js` | Закомментированный код и `console.log` statements | `console.log('Reviews fetched:', reviewsResponse.data);` |
| `AuthorsList.js` | Дублирование кода диалогов (Add/Edit) | Диалоги для книг и авторов имеют практически идентичную структуру |
| Бэкенд-логи | Ошибки базы данных не обрабатываются красиво | `PSQLException: Key (user_id)=(5) is not present in table "users"` |

#### <a name="duplicate_code"/>2.2 Дублирующийся код
| Компонент | Проблема |
| :--- | :--- |
| Фронтенд | Функция `getFullName` дублируется в `AuthorsList.js`, `BooksList.js`, `BookDetails.js`, `Home.js` |
| Фронтенд | Диалоги добавления/редактирования книг и авторов имеют ~80% одинакового кода |
| Бэкенд | Вероятно дублирование логики валидации в контроллерах и сервисах (нужно проверить) |

#### <a name="lack_automation"/>2.3 Отсутствие автоматизации (тестов, сборки, развёртывания)
| Область | Проблема |
| :--- | :--- |
| Тесты | В проекте полностью отсутствуют автоматические тесты (unit, integration, acceptance) |
| Сборка | Нет автоматизации сборки (CI/CD) |
| Развёртывание | Деплой происходит вручную (судя по логам, запуск через IntelliJ) |

#### <a name="bad_architecture"/>2.4 Запутанная архитектура и ненужные сложные зависимости
| Компонент | Проблема |
| :--- | :--- |
| Фронтенд | Смешение бизнес-логики и UI-компонентов (например, прямые `axios` вызовы внутри компонентов) |

#### <a name="slow_tools"/>2.5 Медленные / неэффективные средства
| Проблема | Описание |
| :--- | :--- |
| Ручное тестирование | Отсутствие автотестов приводит к необходимости ручного тестирования каждого изменения |
| Долгий запуск | Spring Boot с DevTools перезапускается при каждом изменении (12+ секунд из логов) |
| Нет ленивой загрузки | На фронтенде все компоненты загружаются сразу (можно оптимизировать) |

#### <a name="uncommitted_code"/>2.6 Незакоммиченный код / долгоживущие ветки
| Проблема | Описание |
| :--- | :--- |
| Закомментированный код | В `BookDetails.js` есть закомментированные блоки и `console.log` |
| Временные решения | Хардкодный админ — классическое временное решение, которое осталось в коде |

#### <a name="lack_docs"/>2.7 Отсутствие / несоответствие технической документации
| Документ | Статус |
| :--- | :--- |
| `README.md` | Отсутствует (нет инструкций по установке, запуску, настройке) |
| API документация | Нет Swagger/OpenAPI спецификации |
| Комментарии в коде | Минимальны, в основном на русском языке (смешение языков) |

#### <a name="lack_test_env"/>2.8 Отсутствие тестовой среды
| Проблема | Описание |
| :--- | :--- |
| Одна среда | Разработка и тестирование ведутся в одной среде (`localhost`) |
| Нет staging | Отсутствует предпродакшн окружение для тестирования |
| Проблемы с БД | Ошибка `user_id=5 not present` могла быть обнаружена на тестовой среде |

#### <a name="long_integration"/>2.9 Длинные циклы интеграции / отсутствие непрерывной интеграции
| Проблема | Описание |
| :--- | :--- |
| Ручная интеграция | Нет CI сервера (Jenkins, GitHub Actions) |
| Интеграция "в конце" | Проблемы интеграции обнаруживаются только при ручном тестировании |

### <a name="action_plan"/>3. План мероприятий по устранению долга
План составлен с учётом 80/20 правила (достижение качества 4/5, а не идеала) и концепции Definition of Done.

#### <a name="priority_critical"/>3.1 Приоритет 1: Критические проблемы (необходимо сделать в ближайший спринт)
| Задача | Описание | Ожидаемый результат |
| :--- | :--- | :--- |
| **1.1 Вынести общие функции** | Создать `utils/helpers.js` с функцией `getFullName` и использовать во всех компонентах | Устранение дублирования |
| **1.2 Добавить базовые acceptance тесты** | Реализовать хотя бы 2-3 ключевых сценария (логин, просмотр книг, добавление книги админом) по методологии ATDD | Возможность автоматической проверки основного функционала |

#### <a name="priority_high"/>3.2 Приоритет 2: Важные улучшения (следующие 1-2 спринта)
| Задача | Описание | Ожидаемый результат |
| :--- | :--- | :--- |
| **2.1 Рефакторинг диалогов** | Создать переиспользуемые компоненты для форм добавления/редактирования книг и авторов | Сокращение дублирования кода на 70% |
| **2.2 Настроить CI (GitHub Actions)** | Добавить базовый pipeline: сборка, прогон тестов, линтинг | Автоматизация проверки качества |
| **2.3 Обработка ошибок** | Добавить глобальный обработчик ошибок на бэкенде и убрать `console.log` на фронтенде | Чистый код и понятные сообщения об ошибках |

#### <a name="priority_medium"/>3.3 Приоритет 3: Желательные улучшения (по мере возможности)
| Задача | Описание | Ожидаемый результат |
| :--- | :--- | :--- |
| **3.1 Создать staging окружение** | Развернуть копию приложения на тестовом сервере (например, на Render.com) | Возможность тестирования в изолированной среде |
| **3.2 Ленивая загрузка** | Добавить `React.lazy()` для маршрутов | Ускорение начальной загрузки |
| **3.3 Улучшить логирование** | Настроить структурированное логирование на бэкенде (например, Logstash) | Упрощение отладки |

### <a name="comparison"/>5. Сравнение объёма долга и недоимплементированных фич

**Текущее состояние:**
- ✅ Авторизация работает
- ✅ Просмотр книг/авторов доступен
- ✅ Отзывы работают
- ✅ Админ может добавлять/редактировать (после исправлений)

**Недоимплементированные фичи (бэклог продукта):**
- Поиск по книгам с фильтрацией
- Пагинация на бэкенде 
- Возможность загружать обложки книг 
- Личный кабинет пользователя
- Рекомендательная система

**Сравнение:**
| Аспект | Объём (в днях) |
| :--- | :--- |
| Технический долг (критичный) | 2 дня |
| Технический долг (полный план) |  5 дней |
| Новая функциональность (примерно) | 7 дней |

**Вывод:** Технический долг составляет **~20%** от объёма новой функциональности. Критическая часть долга — **~13%**. Это значительная, но не катастрофическая величина.

### <a name="conclusions"/>6. Выводы и коррективы в план разработки

**Основные выводы:**
1.  **Долг существует и мешает** — особенно проблемы с авторизацией (хардкод) и целостностью БД уже проявляются в виде ошибок.
2.  **Критический долг требует немедленного вмешательства** — без исправления задач 1.1-1.4 проект остаётся ненадёжным.
3.  **Правило "сломанных окон"** — наличие временных решений (хардкодный админ) провоцирует появление нового некачественного кода.
4.  **80/20 правило применимо** — исправление топ-5 проблем (1.1-2.1) устранит 80% негативных последствий.

**Коррективы в план разработки:**
- **Заморозить** разработку новых фич на **1 спринт**.
- Посвятить этот спринт полному устранению проблем **Приоритета 1**.
